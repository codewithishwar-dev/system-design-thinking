# SOLID Principles — Explained Simply 🚀

Nobody teaches you this in college, but every senior developer swears by it.

## S — Single Responsibility Principle

One class, one job.
Avoid creating “God classes” that handle everything.

## O — Open/Closed Principle

Add new features without modifying existing code.
Good systems grow through extension, not constant rewrites.

## L — Liskov Substitution Principle

Child classes should behave like their parent classes.
If a Penguin throws a “can’t fly” exception, the design is broken.

## I — Interface Segregation Principle

Keep interfaces small and focused.
A Robot should not implement `eat()` or `sleep()`.

## D — Dependency Inversion Principle

Depend on abstractions, not concrete implementations.
Switch databases or services without rewriting your business logic.

Once you start thinking in SOLID, you stop writing code that only works today and start building software that lasts.
